1. Architectural Overview & System Design

The Login Module serves as the security gateway between the user interface layers (React frontend blueprints built from Figma) and the underlying server infrastructure (PHP engine and MySQL relational storage layers). It controls state distribution based on role verification.

Role Identification & System Routing

Upon validation of credentials, the backend session manager executes the Identify Role function. This module acts as an architecture switchboard:

  • Admin Role Authorization: Redirects the user session token to the comprehensive desktop Admin Portal Dashboard to handle properties, contracts, and analytics.
  • Tenant Role Authorization: Restricts the session scope and seamlessly adapts the client view to the responsive mobile Tenant Portal interface for profile tracking, repair logs, and billing.

3. Step-by-Step Sequential Logic Flows

A. Authentication & Role Identification Engine

The following pseudocode sequence executes on the PHP application server layer upon login button interaction:

START Authentication_Process(input_user, input_password) IF input_user IS EMPTY OR input_password IS EMPTY THEN RETURN Error("All fields must be completed.") ENDIF Fetch user_record FROM database WHERE username == input_user OR email == input_user IF user_record NOT FOUND THEN RETURN Error("Invalid username or password.") ENDIF IF user_record.failed_attempts >= 5 THEN SET user_record.account_status = "Locked" RETURN Error("Account locked due to excess failures. Contact Admin.") ENDIF IF Verify_Hash(input_password, user_record.password_hash) == TRUE THEN SET user_record.failed_attempts = 0 INITIALIZE Session(user_record.id, user_record.role) IF user_record.role == "Admin" THEN REDIRECT TO "/admin/dashboard" ELSE IF user_record.role == "Tenant" THEN REDIRECT TO "/tenant/portal" ENDIF ELSE INCREMENT user_record.failed_attempts BY 1 UPDATE database.user_record RETURN Error("Invalid username or password.") ENDIF END

B. 5-Step Password Recovery & Token Validation Flow

Handles self-service identity updates without giving up administrator control boundaries:

START Password_Recovery_Flow(input_email) // Step 1: User Request & Email Verification Fetch account WHERE email == input_email IF account NOT FOUND THEN RETURN Error("Invalid username or password.") // Masking account absence ENDIF // Step 2: Generation of Secure Token SET recovery_token = Generate_Secure_Random_String(32) SET token_expiration = Current_Time() + 15 Minutes SAVE recovery_token, token_expiration TO database.account // Step 3: Dispatch Verification Token Execute Send_System_Email(input_email, recovery_token) DISPLAY "A verification pathway link has been dispatched to your email." // Step 4: Validate Expiration Boundaries & Token Identity LISTEN FOR User_Reset_Submission(submitted_token, new_password) IF Current_Time() > token_expiration THEN RETURN Error("The security token window has expired. Please restart.") ENDIF // Step 5: Commit Structural Password Reset Update IF Validate_Password_Complexity(new_password) == TRUE THEN SET account.password_hash = Hash_Password(new_password) SET account.recovery_token = NULL SET account.failed_attempts = 0 UPDATE database.account REDIRECT TO "/login" WITH Success_Message("Password reset successfully.") ELSE RETURN Error("Password fails system complexity guidelines.") ENDIF END
